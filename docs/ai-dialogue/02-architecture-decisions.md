# 架构决策记录

## 决策 1：为什么用 Next.js App Router 而不是 Vite + React？

**时间**：2026-05-08

**背景**：需要选择一个全栈框架，支持 AI API 调用（需要后端代理，不能暴露 API Key）

**选项**：
- Vite + React（纯前端）→ 不行，AI API Key 会暴露
- Express.js + React（前后端分离）→ 可以，但部署麻烦
- Next.js App Router（全栈）→ **选中**

**AI 的建议**：
> "如果你要展示全栈能力，Next.js App Router 是最好的选择。Server Actions 可以天然隐藏 API Key，而且 RSC 可以减少客户端 bundle。"

**我的判断**：同意，而且 Next.js 在求职市场更常见，HR 看到会更有认知。

## 决策 2：AI 输出要不要加推理链展示？

**背景**：用户提出"AI 黑盒"问题 —— 如果 AI 直接给答案，用户不信任。

**我的方案**：在 PRD 生成结果里，展示 AI 的"思考过程"：
```
【AI 的思考】
1. 识别核心问题：用户描述的是"任务管理"，但深层需求是"减少决策疲劳"
2. 用户画像推断：独立开发者，技术能力强但产品感弱
3. 功能优先级判断：先解决"快速记录"，再解决"智能分类"
...
```

**AI 的反对意见**：
> "展示推理链会增加 token 消耗（约 30%），而且普通用户可能不关心。建议做成可折叠的'高级模式'。"

**最终决策**：听取 AI 建议，做成可折叠的。展示推理链是为了**给面试官看**，不是给真实用户看。

## 决策 3：任务拆解要不要接入 Jira / Notion API？

**我的初始想法**：做集成，直接推任务到 Jira，显得功能强大。

**AI 反问**：
> "你有多少用户真的会用这个功能？大多数独立开发者用 Notion 或 GitHub Issues，不是 Jira。而且接 Jira API 需要 OAuth，开发量很大，值不值得？"

**我的反思**：AI 说得对。这个功能看起来酷，但实际使用频率低，开发成本高。

**最终决策**：只做"导出 Markdown"，用户自己复制粘贴。MVP 原则。

## 决策 4：技术选型推荐，依据是什么？

**问题**：如果 AI 推荐的技术栈是"拍脑袋"的，那这个产品没有价值。

**我的方案**：技术选型推荐必须附带：
1. 该技术的 GitHub Stars / 最近更新时间（证明不是过时技术）
2. 与需求的匹配度评分（为什么选 A 不选 B）
3. 学习曲线评估（新手是否友好）

**AI 的建议**：
> "你没有办法实时获取 GitHub Stars（除非接 GitHub API）。建议在 prompt 里让 AI 基于'2024-2026 年流行度'来推荐，并加上免责声明。"

**最终方案**：
- Prompt 里明确限定"2024-2026 主流技术"
- 输出格式强制要求"为什么不选 XXX"
- 加免责声明："技术选型建议仅供参考，请结合团队情况判断"
