# Prompt 工程迭代记录

## 问题：初版 Prompt 输出质量不稳定

### V1（失败）

```
你是一个产品经理，帮我生成 PRD。
```

**输出的问题**：
- 内容太泛，没有针对性
- 格式混乱，有时会输出 JSON，有时会输出 Markdown
- 没有优先级判断，所有功能平铺

---

### V2（改进：加格式约束）

```
你是一位高级产品经理。请生成 PRD，格式为 Markdown，包含：
1. 问题定义
2. 用户画像
3. 功能列表
```

**输出的问题**：
- 格式稳定了，但内容还是泛
- "用户画像"永远是"25-35岁白领"，明显是模板

---

### V3（改进：加推理链）

```
你是一位高级产品经理。

第一步：先思考（但不要输出）用户描述背后的深层需求是什么。
第二步：基于深层需求，生成 PRD。

输出格式：
## 深层需求分析
## 问题定义（Jobs-to-be-Done 框架）
## 用户画像（3 类，每类必须有具体场景）
## 功能列表（MoSCoW 优先级）
```

**效果**：明显变好，"深层需求分析"部分能抓到用户没说出来的东西。

---

### V4（改进：加反面思考）

```
...（前面同上）

在生成功能列表时，请先列出"不应该做的功能"，避免范围蔓延。
```

**效果**：AI 开始主动做减法，PRD 更聚焦。

---

### V5（最终版：加输出长度约束 + 示例）

```
你是一位有 10 年经验的高级产品经理，擅长 MVP 设计。

## 任务
根据用户描述，生成结构化 PRD。

## 输出要求
1. 每个 section 不超过 200 字（强迫优先级判断）
2. 用户画像必须有"使用场景"，不能只有人口属性
3. 功能列表用 MoSCoW 分类，Must-have 不超过 5 个
4. 先输出"我们不做什么"，再做功能列表

## 格式
Markdown，用二级标题（##）分节。

## 示例（输入："我想做一个任务管理 App"）
输出：
### 深层需求：用户不是要"管理任务"，而是"减少决策疲劳"
### 问题定义：...
```

**效果**：稳定输出高质量 PRD。

---

## 关键学习

1. **约束比自由更重要** —— 限制输出长度，AI 反而更好
2. **示例驱动** —— 给一个完整示例，比写 10 条规则更有效
3. **先否定再肯定** —— "不做什么"比"做什么"更能聚焦
4. **推理链要隐藏** —— 让 AI "先思考"但不输出思考过程，直接给结果，结果质量更高

## 这个文档的意义

展示我对 Prompt Engineering 的理解：**不是"会写 Prompt"，而是"理解 AI 的认知模式，设计能让它发挥最大能力的框架"**。

这才是真正的能力。
